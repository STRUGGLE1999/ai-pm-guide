# CH1 模型部署基础

模型部署是把训练好或选好的模型变成可被业务系统调用的服务。Demo 阶段只要能跑，生产阶段要考虑延迟、并发、稳定性、日志、权限、成本和回滚。

## 部署方式

| 方式 | 适合场景 |
| --- | --- |
| 云 API | 快速验证、低敏场景。 |
| 私有化部署 | 数据敏感、内网、合规要求高。 |
| 混合部署 | 按数据等级和任务复杂度分流。 |
| 本地小模型 | 离线、端侧、低成本固定任务。 |

## 技术要点

1. 模型服务部署在哪里？
2. 谁负责运维和升级？
3. 峰值并发多少？
4. 是否需要流式输出？
5. 出错后怎么降级？
6. 日志是否脱敏？

部署不是技术细节，它决定 AI 功能能不能稳定进入真实业务。

## 从 Demo 到生产缺什么

| Demo 阶段 | 生产阶段 |
| --- | --- |
| 手动运行脚本 | 服务自动启动和健康检查。 |
| 单人测试 | 多用户并发和限流。 |
| 少量样例 | 真实数据和边界样本。 |
| 没有日志 | 可查询、可脱敏、可审计日志。 |
| 模型固定 | 版本、灰度、回滚。 |
| 成本无感 | Token、GPU、缓存、人工复核成本。 |

很多 AI 项目演示顺利，上线困难，是因为 Demo 只证明“能回答”，没有证明“能稳定服务”。

## 部署验收清单

- 服务能否自动恢复？
- 是否有健康检查接口？
- 是否记录模型版本和配置版本？
- 是否能按用户、任务、时间查询日志？
- 并发压测是否通过？
- 模型不可用时是否有降级方案？
- 高风险输出是否有人审？

## 工程实践要点

部署与加速决定 AI 能力能否被真实用户持续使用。Demo 阶段只要能跑，生产阶段要看并发、延迟、成本、稳定性、安全和回滚。读者需要把“模型效果”扩展为“完整链路体验”。

| 指标 | 含义 | 工程影响 |
| --- | --- | --- |
| TTFT | 首个 Token 出现时间。 | 影响聊天等待感。 |
| 总延迟 | 完整任务完成时间。 | 影响用户是否愿意继续用。 |
| 吞吐 | 单位时间处理多少请求。 | 影响高峰期稳定性。 |
| 成本 | 单次任务消耗多少钱。 | 影响商业模式是否成立。 |
| 成功率 | 请求是否稳定完成。 | 影响信任和 SLA。 |

## 示例配置

下面是一个最小模型服务接口示例，用 FastAPI 暴露推理入口：

```python
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

class PredictRequest(BaseModel):
    text: str

@app.post("/predict")
def predict(req: PredictRequest):
    return {
        "label": "example",
        "score": 0.98,
        "input_length": len(req.text),
    }
```

真实部署需要补充模型加载、批处理、超时、日志、鉴权、限流、健康检查和版本管理。

## 技术细节拆解

CH1 模型部署基础 可以拆成输入、处理、输出和指标四层。这样读的时候不会停留在概念名，而是能看到它在系统里接收什么、改变什么、产出什么。

| 层次 | 具体内容 |
| --- | --- |
| 输入 | 模型文件、tokenizer、配置、请求参数、运行硬件。 |
| 处理 | 预处理、推理、后处理、缓存、限流、监控。 |
| 输出 | 接口响应、状态码、日志、指标、告警。 |
| 指标 | TTFT、总延迟、QPS、P95/P99、显存占用、错误率。 |

## 关键参数和边界

| 参数/边界 | 说明 |
| --- | --- |
| 并发 | 并发过高会排队，延迟上升。 |
| batch | 合批能提高吞吐，但可能增加单个请求等待时间。 |
| 量化 | 降低显存和成本，但可能损失精度。 |
| 超时 | 所有外部调用都要设置超时和降级。 |

## 可运行检查

```python
import time

start = time.perf_counter()
# result = model.generate(inputs)
time.sleep(0.05)
latency_ms = (time.perf_counter() - start) * 1000
print({"latency_ms": round(latency_ms, 2)})
```

## 怎么判断学懂了

| 判断点 | 具体标准 |
| --- | --- |
| 延迟分解 | 分别统计预处理、检索、推理、后处理耗时。 |
| 资源曲线 | 观察 CPU、内存、GPU、显存随并发变化。 |
| 回滚验证 | 保留旧版本模型和配置，演练故障回退。 |

## 常见误区和排查

| 问题 | 为什么会发生 | 怎么排查 |
| --- | --- | --- |
| 延迟只看平均值 | P95/P99 很差会被平均值掩盖。 | 压测时同时记录平均、P95、P99。 |
| 无超时和降级 | 外部依赖卡住导致请求堆积。 | 每个外部调用设置 timeout 和 fallback。 |
| 版本不可回滚 | 模型、配置、索引和 Prompt 没绑定版本。 | 建立发布清单和回滚脚本。 |
